# NekoProtoTools 命名规范

## 目标

命名首先服务用户读写 API 的体验，其次服务维护者阅读实现时的统一性：

1. 用户侧 DSL 自然、短、好读。
2. 同一概念只有一个正式名字，不保留兼容别名。
3. 模板元编程部分接近 STL 风格，减少 C++ 用户的认知负担。
4. clang-tidy 能稳定执行，而不是靠大量 `NOLINT` 压过去。

本项目不按“是否为模板”决定命名风格。真正的分类依据是：

```text
公开概念类型      -> CamelCase
元编程工具        -> lower_case
type-level key    -> lower_case
函数              -> lower_case
变量 / 成员       -> lower_case
宏                -> UPPER_CASE
```

## 总规则

公开概念类型使用 `CamelCase`。这类类型代表库的正式概念，即使它们是模板，也使用首字母大写的驼峰：

```cpp
JsonTag
BinaryTag
TaggedField
TagList
NoTags
Object
Array
Enumerate
Reflect
Parser
Result
Error
```

trait、metafunction、alias template、variable template 使用 STL 风格的 `lower_case`：

```cpp
is_tag_list
is_tagged_field
unwrap_tagged_field
normalize_tags
resolve_member_type
tag_list_element

is_tag_list_v
is_tagged_field_v
unwrap_tagged_field_t
unwrap_tagged_field_tags_v
normalize_tags_v
resolve_member_type_t
field_tags_v
```

type-level query key 使用 `lower_case`，统一放在 `tag_property` 命名空间下：

```cpp
namespace tag_property {

struct comment;
struct name;
struct skippable;
struct raw_string;

template <typename Value>
struct flat;

template <typename Value>
struct unframed;

template <typename Value>
struct fixed_length;

} // namespace tag_property
```

函数、成员函数、静态成员函数、constexpr 函数使用 `lower_case`：

```cpp
make_tags(...)
field_accessor(...)
tag_query::get<...>(...)
tag_query::has<...>(...)
tag_query::get_tag<...>(...)
```

变量、参数、公有成员、私有成员都使用 `lower_case`。私有成员使用后缀 `_`：

```cpp
auto field_name = ...;
auto tag_count = ...;

class Reader {
private:
    std::string_view input_;
    std::size_t cursor_ = 0;
};
```

用户侧 aggregate DSL 的字段也必须是 `lower_case`：

```cpp
JsonTag{.skippable = true, .raw_string = true}
BinaryTag{.fixed_length = 4, .unframed = true}
ArgTags{.range_min = 1, .range_max = 65536}
```

模板参数使用 `CamelCase`，传统单字母参数可以保留：

```cpp
template <typename Accessor, typename HostType>
struct resolve_member_type;

template <typename T, typename U>
inline constexpr bool same_as_v = ...;
```

公开宏使用 `NEKO_` 前缀，内部宏使用 `NEKO_DETAIL_` 前缀：

```cpp
NEKO_SERIALIZER
NEKO_BEGIN_NAMESPACE
NEKO_END_NAMESPACE
NEKO_DETAIL_DEFINE_TAG_PROPERTY
NEKO_DETAIL_DEFINE_TYPE_TAG_PROPERTY
```

## Tags API

用户侧推荐写法：

```cpp
make_tags<
    comment_tag<"visible in schema">,
    rename_tag<"wire_name">,
    JsonTag{.skippable = true}
>(&User::name);
```

二进制布局 tag：

```cpp
make_tags<
    BinaryTag{.fixed_length = 4, .unframed = true}
>(&Header::length);
```

raw JSON 字符串：

```cpp
make_tags<
    JsonTag{.raw_string = true}
>(&Message::payload);
```

查询侧：

```cpp
constexpr auto tags = field_tags_v<SomeField>;

static_assert(tag_query::has<tag_property::name>(tags));
static_assert(tag_query::get<tag_property::name>(tags) == "wire_name");

constexpr bool skippable =
    tag_query::get<tag_property::skippable>(tags);
```

按 tag 类型查询时使用 `has_tag` / `get_tag`：

```cpp
static_assert(tag_query::has_tag<JsonTag>(tags));
constexpr auto json_tag = tag_query::get_tag<JsonTag>(tags);
```

模块内部实现放入各自模块的 `detail`。跨 serialization / argparser / rpc 共享的 tag 基础设施放入
`tag_detail`，避免在 `argparser::detail` 等嵌套命名空间内产生歧义。用户 API 只暴露变量模板和公开概念类型：

```cpp
comment_tag<"...">
rename_tag<"...">
JsonTag{...}
BinaryTag{...}
```

## 执行边界

同一概念只保留一个名字。命名重构落地时，需要同时更新实现、测试、README、当前规范文档和示例代码；不通过别名制造迁移窗口。

历史设计文档可以保留演进背景，但当前推荐写法必须与代码一致。面向用户的文档、测试用例和 README 不应再展示被替换的写法。

`.clang-tidy` 是可执行约束来源；本文档解释命名意图，具体检查项以根目录配置为准。
